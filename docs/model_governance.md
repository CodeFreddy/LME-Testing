# Model Governance

## Purpose

This document defines how the repository governs model usage across maker, checker, rewrite, planner, and future AI-assisted modules.

The goal is to ensure that introducing or changing LLM APIs does not silently degrade artifact quality, traceability, or release stability.

This document applies to:

- model providers,
- model versions,
- prompts,
- structured output contracts,
- benchmark and regression workflows,
- rollout and rollback decisions.

This document should be used together with:

- `docs/roadmap.md`
- `docs/acceptance.md`
- `docs/architecture.md`
- `docs/agent_guidelines.md`

---

## Governance Objectives

Model governance exists to enforce the following outcomes:

1. model changes are visible and auditable,
2. prompt changes are versioned and reviewable,
3. artifact quality is benchmarked before adoption,
4. model-specific behavior is isolated from business logic,
5. rollouts are reversible,
6. stable development remains possible across multiple model APIs.

---

## Core Governance Principles

### 1. Treat model output as probabilistic

All model outputs are probabilistic and must be constrained by:

- schemas,
- validation gates,
- benchmark checks,
- metadata,
- rollback-safe rollout procedures.

### 2. Keep business logic model-agnostic

Business logic must not depend on undocumented behavior of a specific model vendor.

Any provider-specific handling must remain behind a provider or strategy interface.

### 3. Prefer structured outputs over free-form text

If a module can emit JSON, it must emit JSON.

Structured output contracts are part of the repository's stability guarantees.

### 4. No silent model adoption

A model must not become the default provider for any production workflow unless it passes the defined compatibility and benchmark gates.

### 5. Every model change must be attributable

Every artifact generated by a model-backed workflow must record enough metadata to explain how it was produced.

This policy applies to all AI-assisted modules, including but not limited to:

- maker
- checker

It also applies to:

- provider configuration,

## Required Provider Abstraction

All model integrations must conform to a stable provider abstraction.

At minimum, the provider contract must expose:

- provider name
- model name
- model version if available
- API base or provider identity
- retry policy
- timeout policy
- decoding or generation settings
- structured output mode if supported
- error classification behavior

The provider layer must isolate:

- vendor-specific request formats,
- vendor-specific structured output quirks,
- retry semantics,
- transport-specific errors.

Business modules such as maker or checker must not embed vendor-specific request logic directly.

## Prompt Governance

### Prompt Identity

Every production prompt must have:

- a stable prompt ID,
- semantic version,
- module owner,
- intended module (`maker`, `checker`, `rewrite`, etc.),
- linked benchmark set,
- change log entry.

### Prompt Storage

Prompts should be stored in a discoverable repo location such as:

- `prompts/`
- `docs/prompts/`
- or another clearly documented prompt directory.

Prompt templates must not be hidden inside unrelated code paths unless necessary, and if embedded they must still expose version metadata.

### Prompt Change Policy

Any prompt change that can affect output behavior must include:

- version bump,
- reason for change,
- expected impact,
- benchmark comparison,
- reviewer approval.

Minor wording changes that do not affect semantics may use a patch version. Behavioral or structural changes should use a minor or major version depending on impact.

### Prompt Version in Artifacts

Any artifact produced by a model-backed workflow must record:

- prompt ID,
- prompt version,
- model ID,
- provider,
- run timestamp,
- source artifact hash,
- pipeline version.

---

## Artifact Metadata Requirements

Artifacts generated by maker, checker, rewrite, planner, and future model-assisted stages must include machine-readable metadata.

Required fields:

- `provider`
- `model_id`
- `prompt_id`
- `prompt_version`
- `run_timestamp`
- `pipeline_version`
- `source_artifact_hash`

Recommended additional fields:

- generation settings summary,
- retry count,
- structured output enforcement mode,
- benchmark tag if run under evaluation mode.

Metadata should be retained through downstream stages where practical.

## Model Compatibility Policy

A model may be considered compatible only if it satisfies all required checks.

### Required Compatibility Checks

1. **Schema conformance**
   - outputs validate against required schemas,
   - invalid or partial outputs are surfaced correctly.

2. **Structural stability**
   - repeated runs do not frequently break required structural fields,
   - JSON shape remains usable across repeated executions.

3. **Benchmark threshold**
   - model performance meets or exceeds the required baseline threshold for the target module.

4. **Checker stability threshold**
   - for checker-like modules, repeated evaluation must remain within the defined instability budget.

5. **Artifact diff review**
   - meaningful differences versus the current baseline are reviewed before adoption.

6. **Failure-mode acceptability**
   - transport, timeout, truncation, and malformed-output behaviors are operationally acceptable.

### Compatibility Outcome Levels

A model should be classified as one of:

- `experimental`
- `candidate`
- `approved`
- `deprecated`
- `blocked`

Only `approved` models may be used as defaults for production workflows.

## Benchmark Policy

### Benchmark Purpose

Benchmarks exist to prevent silent degradation when:

- switching provider,
- upgrading model version,
- changing prompts,
- modifying structured output contracts.

### Benchmark Categories

At minimum, the repository should maintain benchmark coverage for:

1. extraction sanity
2. maker structural validity
3. checker consistency
4. report generation compatibility
5. end-to-end smoke flow

As the platform matures, benchmarks should expand to include:

- document-class-specific extraction quality,
- planner output quality,
- BDD normalization quality,
- step mapping quality,
- execution readiness quality.

### Baseline Dataset

A small official baseline must exist for fast regression checks.

The existing minimal sample flow should be used as the initial baseline seed, then expanded into:

- smoke baseline,
- stable regression baseline,
- model evaluation baseline.

### Benchmark Rules

- benchmark datasets must be versioned,
- benchmark outputs must be comparable over time,
- benchmark results must be stored or exportable,
- benchmark thresholds must be module-specific,
- benchmark failures must block model promotion.

---

## Stability and Consistency Policy

### Maker Stability

Maker runs must be evaluated for:

- schema validity rate,
- required case type completeness,
- duplicate scenario rate,
- artifact diff against baseline.

### Checker Stability

Checker runs must be evaluated for:

- repeated-run consistency,
- decision drift,
- coverage conclusion drift,
- parse and schema validity.

Checker instability should be surfaced explicitly, not hidden behind final coverage labels.

### Planner Stability (Future)

Planner-like modules should be evaluated for:

- reproducibility of planning structure,
- schema validity,
- traceability completeness,
- risk and priority output consistency.

## Rollout Policy

### Rollout Stages

Any new model or prompt should move through the following stages:

1. **Experimental**
   - limited local or benchmark-only use,
   - not used for default production runs.

2. **Candidate**
   - benchmark-passing,
   - under controlled comparison against the current approved baseline.

3. **Approved**
   - accepted for default use in defined modules,
   - documented in the approved model list.

4. **Deprecated**
   - still supported temporarily,
   - planned for removal.

5. **Blocked**
   - must not be used for target modules.

### Promotion Requirements

A model or prompt may be promoted only if:

- benchmark thresholds pass,
- schema conformance is stable,
- known failure modes are documented,
- rollback path exists,
- artifact diffs are reviewed.

---

## Rollback Policy

Every production model or prompt change must have a rollback path.

Rollback must be possible without:

- changing business logic,
- rewriting artifact schemas,
- invalidating stored artifacts.

Rollback readiness requires:

- previous approved model still available, or
- equivalent fallback model approved,
- prompt versions retained,
- provider configuration reversible,
- migration notes if compatibility changed.

Rollback triggers may include:

- benchmark regression,
- rising checker instability,
- malformed artifact increase,
- provider outage or degradation,
- unacceptable semantic drift.

---

## Release Gate for Model Changes

Any release that changes model behavior must satisfy the following before merge or release:

- schemas validate,
- benchmark passes,
- compatibility outcome documented,
- artifact metadata present,
- prompt version updated if applicable,
- rollback path documented,
- reviewer approval recorded.

Recommended additional checks:

- artifact diff snapshot,
- checker repeated-run comparison,
- known failure modes note,
- release note summary.

## Failure Mode Policy

Known model and provider failure modes must be documented.

Examples include:

- malformed JSON,
- missing required fields,
- truncation,
- timeout,
- retry exhaustion,
- inconsistent checker decisions,
- vendor-specific structured output edge cases.

For each known failure mode, define:

- detection method,
- severity,
- mitigation,
- whether it blocks release,
- whether it triggers rollback.

## AI Agent Rules for Model-Related Changes

Any AI coding agent modifying model-backed workflows must follow these rules.

### 1. Do not hardcode vendor-specific logic in business modules

Provider-specific handling belongs in provider or adapter layers only.

### 2. Do not change prompts without versioning

Every prompt-affecting change must update prompt version metadata and related docs.

### 3. Do not declare a model “better” without benchmark evidence

Claims about quality improvements must include benchmark or stability evidence.

### 4. Do not remove structured output validation

Any relaxation of validation rules must be explicitly approved and documented.

### 5. Do not switch defaults silently

A default model change must update:

- approved registry,
- benchmark references,
- rollout docs,
- rollback notes.

## Maintenance Rules

This document should be updated whenever:

- a new provider is adopted,
- a new model becomes approved,
- a prompt versioning policy changes,
- benchmark thresholds change,
- rollout or rollback policy changes,
- structured output strategy changes.

The governance owner should periodically review:

- approved model status,
- deprecated model removal,
- benchmark drift,
- checker instability trends,
- provider outage or error trends.

---

## Summary

Model governance is not a documentation extra. It is a core engineering requirement for a multi-model AI testing framework. This repo must treat models, prompts, and structured outputs as governed dependencies. Stable provider abstraction, prompt versioning, artifact metadata, benchmark gates, rollout stages, and rollback rules are required to ensure that model changes do not silently degrade testing artifacts or roadmap execution.
