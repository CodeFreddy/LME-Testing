**批判性架构审查**

## 1. 一句话定位

这个 repo 本质上是一个**文档驱动的测试设计与审核流水线原型**：它把 LME 文档抽成 `atomic_rule` / `semantic_rule`，再用 maker 生成 BDD 风格场景、checker 做覆盖审查、人工 review 决定 rewrite/approve/reject，并输出 HTML 报告；它**不是**一个完整的测试执行系统，也不是成熟的测试平台。

## 2. 已落地能力

仓库里已经清楚落地的能力有这几类。

第一，**规则抽取与归一化**。README 和设计文档都把处理链路写得很明确：从源文档抽规则，生成 `atomic_rule` 和 `semantic_rule`，并强调 `atomic_rule` 是文档事实层、`semantic_rule` 是语义层和“半执行层”。文档还明确给出推荐流水线是 `PDF -> pages -> clauses -> atomic_rule -> semantic_rule -> Gherkin/DSL -> review -> coverage`。

第二，**maker / checker / rewrite 流水线**。README、usage 文档和 `pipelines.py` 都表明系统已经实现 maker、checker、rewrite 这几个核心环节；其中 maker 会基于 `semantic_rules.json` 生成结构化场景，checker 会按 case 级别输出结构化评审，rewrite 只重生成被人工标记为 `rewrite` 的规则。

第三，**严格覆盖规则**。这个 repo 不是随便生成几个 case 就算完成。`pipelines.py` 里定义了 `RULE_TYPE_CASE_REQUIREMENTS`，把 `obligation`、`deadline`、`workflow` 等不同 `rule_type` 映射到各自 required/optional case types；README 和 usage 文档也明确说，只有所有 required case types 都被 checker 接受，规则才算 fully covered。

第四，**模型输出约束与回流校验**。`schemas.py` 里对 maker、checker、human review 都做了严格结构校验，包括：maker 必须对每个输入 rule 返回恰好一个结果、每个 required case type 恰好一个 scenario、evidence 要短且单行；checker 必须对每个输入 case 返回恰好一个结果，且 `semantic_rule_id` 必须与输入映射一致。

第五，**人工审核与本地 review session**。README 和 usage 文档写明 `review-session` 是主要人工评审入口，支持保存草稿、提交 review、执行 `rewrite -> checker -> report`、多轮 review/rewrite cycle，以及最终跳转到 `report.html`。人工 review 字段和枚举也被固定成结构化格式，而不是完全自由文本。

第六，**基础可靠性措施**。provider 配置和 usage 文档显示它有 retry、resume、日志输出这些工程缓冲措施；默认重试参数是 `max_retries = 3`、`retry_backoff_seconds = 2.0`，面向超时、`429/5xx` 和连接中断等可恢复错误。

## 3. 容易被误以为已落地、其实没有的能力

最容易被高估的是：它看起来像“测试自动化框架”，但仓库证据更支持它是**测试设计与审核系统**。README 的主流程停在 extract → maker → checker → review-session → rewrite → report；主要入口也集中在 CLI、pipeline、review session、reporting、provider adapter 这些模块上，没有看到真正的执行引擎、环境编排、驱动器、测试数据准备层、SUT 连接层或断言运行时。

第二个容易被高估的点是 `semantic_rule` 里的 `execution_mapping` 和 `dsl_assertions`。样例 `semantic_rules.json` 里确实已经有这些字段，设计文档也说 `semantic_rule` 要“对未来可执行测试保持足够接近”；但同一份设计文档也明确说它只是“半执行层”，**不直接等于测试脚本**。所以这些字段更像桥接结构和未来对接执行层的接口，不等于仓库里已经有一个可运行的执行框架。

第三个容易被误判的点是“严格 schema = 确定性质量”。仓库确实用了 validator、required case types、expected ids、exactly-one 这类约束；但 maker / checker / rewrite 本身仍然是模型调用流程，provider 也还是 openai-compatible 配置。这意味着它是在**用 schema 把 LLM 输出收紧**，不是把系统变成一个确定性算法系统。

## 4. 三个最大局限

### 局限一：它最大的能力边界，就是“设计/审核”而不是“执行/运行”

这是这个 repo 最核心的边界。现在的闭环是：文档抽取、规则建模、场景生成、覆盖审查、人工评审、重写、报表。它没有把测试真正“跑起来”的执行层闭环展示出来。换句话说，它更像一个 AI 化的测试分析师/测试设计助手，而不是端到端自动化测试框架。

### 局限二：质量控制强依赖 prompt + schema +人审，而不是确定性判定

仓库已经意识到模型不稳定，所以加了严格的 payload 校验、expected ids、case type 精确要求、短 evidence、human review 枚举等机制。问题在于，这些都属于“收紧输出边界”的工程措施，而不是把正确性变成可证明的确定性判断。结果质量仍取决于 maker/checker 的模型理解、提示词稳定性和供应商兼容行为。

### 局限三：抽取链路对输入文档质量依赖强，尤其 PDF 路径更脆

设计文档明确反对“让大模型直接从 PDF 生成测试案例”，理由包括覆盖率分母不稳定、追溯关系会丢、难以区分拆分错误和语义理解错误；抽取指南也把流程拆成 `pages.json`、`clauses.json`、`atomic_rules.json`、`semantic_rules.json`，并提示 Markdown 路径更适合处理，PDF 路径要额外做分页和中间产物检查。这说明仓库自己就把文档抽取当成一个高误差源在防。

## 5. 三个最高风险

### 风险一：LLM 波动会直接影响设计产物和 coverage 结论

maker 和 checker 都是关键路径的一部分，而 provider 配置展示的也是外部模型服务。即使有 `validate_maker_payload` / `validate_checker_payload`，这些校验只能防止“结构错、映射错、缺项多项”，不能保证模型对业务规则的语义理解始终正确。模型升级、prompt 漂移、供应商 JSON 兼容性差异，都可能让相同规则得到不同的场景与评审结果。

### 风险二：人工 review 是质量闭环的一部分，也会成为吞吐瓶颈

仓库没有把“最终正确性”自动化掉，而是明确让 human reviewer 决定 `approve / rewrite / reject`，且只有 `Decision = rewrite` 才会进入 rewrite；review-session 还支持多轮循环。这种设计适合 PoC 和高风险规则审查，但规模一大，人工 review 会成为速度上限和治理成本来源。

### 风险三：项目成熟度仍明显偏原型，长期维护与团队采纳风险高

GitHub 仓库页显示它目前只有 **2 commits**、**0 stars / 0 forks**、**No releases published**，README 还直接推荐先从 `artifacts/poc_two_rules/semantic_rules.json` 这个两条规则样例开始，以便快速验证和避免长时间运行。再加上规则建模文档开头直接写“本文档面向接手本项目的开发人员”，这些信号都更像内部原型/方法验证，而不是成熟可发行项目。

## 6. 成熟度判断

我的判断是：它现在最像**PoC 向内部原型过渡阶段**，还没到“可正式落地的成熟项目”。

支持这个判断的证据比较硬：

* 仓库历史非常短，主页显示只有 **2 commits**。
* 没有 release，主页直接显示 **No releases published**。
* README 推荐从 **poc_two_rules** 开始做验证，而不是从完整规则集或标准化安装/部署入口开始。
* 核心设计文档明显偏“接手说明”和“设计解释”，不是成熟产品文档风格。
* review-session 是**本地 HTTP review service**，更像开发/验证工具，而不是现成团队级审查平台。

所以它更适合“小规模规则验证、方法论探索、内部演示、测试设计流程打样”，不适合未经改造就直接承担团队级、长期运行的测试平台职责。这个结论是基于仓库当前公开内容；至于作者私下是否有未公开执行层或后续 roadmap，仓库本身没有给出足够证据，我不把那部分算进现状判断。

## 7. 最终结论

**它现在更适合方法验证和小范围 PoC，不适合被高估成完整测试执行平台；最大阻碍不是“它做得不聪明”，而是“它目前停在测试设计/审核层，且关键质量仍然受 LLM 与人工 review 双重约束”。**
