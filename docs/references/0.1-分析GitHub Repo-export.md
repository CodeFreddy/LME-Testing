## 1. 一句话定位

这个 repo 本质上是一个**文档驱动的测试设计与审核流水线原型**：它把 LME 文档抽成 `atomic_rule` / `semantic_rule`，再用 maker 生成 BDD 风格场景、checker 做覆盖审查、人工 review 决定 rewrite/approve/reject，并输出 HTML 报告；它**不是**一个完整的测试执行系统，也不是成熟的测试平台。

## 2. 已落地能力

仓库里已经清楚落地的能力有这几类。

第一，**规则抽取与归一化**。README 和设计文档都把处理链路写得很明确：从源文档抽规则，生成 `atomic_rule` 和 `semantic_rule`，并强调 `atomic_rule` 是文档事实层、`semantic_rule` 是语义层和“半执行层”。文档还明确给出推荐流水线是 `PDF -> pages -> clauses -> atomic_rule -> semantic_rule -> Gherkin/DSL -> review -> coverage`。

第二，**maker / checker / rewrite 流水线**。README、usage 文档和 `pipelines.py` 都表明系统已经实现 maker、checker、rewrite 这几个核心环节；其中 maker 会基于 `semantic_rules.json` 生成结构化场景，checker 会按 case 级别输出结构化评审，rewrite 只重生成被人工标记为 `rewrite` 的规则。

第三，**严格覆盖规则**。这个 repo 不是随便生成几个 case 就算完成。`pipelines.py` 里定义了 `RULE_TYPE_CASE_REQUIREMENTS`，把 `obligation`、`deadline`、`workflow` 等不同 `rule_type` 映射到各自 required/optional case types；README 和 usage 文档也明确说，只有所有 required case types 都被 checker 接受，规则才算 fully covered。

第四，**模型输出约束与回流校验**。`schemas.py` 里对 maker、checker、human review 都做了严格结构校验，包括：maker 必须对每个输入 rule 返回恰好一个结果、每个 required case type 恰好一个 scenario、evidence 要短且单行；checker 必须对每个输入 case 返回恰好一个结果，且 `semantic_rule_id` 必须与输入映射一致。

第五，**人工审核与本地 review session**。README 和 usage 文档写明 `review-session` 是主要人工评审入口，支持保存草稿、提交 review、执行 `rewrite -> checker -> report`、多轮 review/rewrite cycle，以及最终跳转到 `report.html`。人工 review 字段和枚举也被固定成结构化格式，而不是完全自由文本。

第六，**基础可靠性措施**。provider 配置和 usage 文档显示它有 retry、resume、日志输出这些工程缓冲措施；默认重试参数是 `max_retries = 3`、`retry_backoff_seconds = 2.0`，面向超时、`429/5xx` 和连接中断等可恢复错误。

## 3. 容易被误以为已落地、其实没有的能力

最容易被高估的是：它看起来像“测试自动化框架”，但仓库证据更支持它是**测试设计与审核系统**。README 的主流程停在 extract → maker → checker → review-session → rewrite → report；主要入口也集中在 CLI、pipeline、review session、reporting、provider adapter 这些模块上，没有看到真正的执行引擎、环境编排、驱动器、测试数据准备层、SUT 连接层或断言运行时。

第二个容易被高估的点是 `semantic_rule` 里的 `execution_mapping` 和 `dsl_assertions`。样例 `semantic_rules.json` 里确实已经有这些字段，设计文档也说 `semantic_rule` 要“对未来可执行测试保持足够接近”；但同一份设计文档也明确说它只是“半执行层”，**不直接等于测试脚本**。所以这些字段更像桥接结构和未来对接执行层的接口，不等于仓库里已经有一个可运行的执行框架。

第三个容易被误判的点是“严格 schema = 确定性质量”。仓库确实用了 validator、required case types、expected ids、exactly-one 这类约束；但 maker / checker / rewrite 本身仍然是模型调用流程，provider 也还是 openai-compatible 配置。这意味着它是在**用 schema 把 LLM 输出收紧**，不是把系统变成一个确定性算法系统。

## 4. 三个最大局限

### 局限一：它最大的能力边界，就是“设计/审核”而不是“执行/运行”

这是这个 repo 最核心的边界。现在的闭环是：文档抽取、规则建模、场景生成、覆盖审查、人工评审、重写、报表。它没有把测试真正“跑起来”的执行层闭环展示出来。换句话说，它更像一个 AI 化的测试分析师/测试设计助手，而不是端到端自动化测试框架。

### 局限二：质量控制强依赖 prompt + schema +人审，而不是确定性判定

仓库已经意识到模型不稳定，所以加了严格的 payload 校验、expected ids、case type 精确要求、短 evidence、human review 枚举等机制。问题在于，这些都属于“收紧输出边界”的工程措施，而不是把正确性变成可证明的确定性判断。结果质量仍取决于 maker/checker 的模型理解、提示词稳定性和供应商兼容行为。

### 局限三：抽取链路对输入文档质量依赖强，尤其 PDF 路径更脆

设计文档明确反对“让大模型直接从 PDF 生成测试案例”，理由包括覆盖率分母不稳定、追溯关系会丢、难以区分拆分错误和语义理解错误；抽取指南也把流程拆成 `pages.json`、`clauses.json`、`atomic_rules.json`、`semantic_rules.json`，并提示 Markdown 路径更适合处理，PDF 路径要额外做分页和中间产物检查。这说明仓库自己就把文档抽取当成一个高误差源在防。

## 5. 三个最高风险

### 风险一：LLM 波动会直接影响设计产物和 coverage 结论

maker 和 checker 都是关键路径的一部分，而 provider 配置展示的也是外部模型服务。即使有 `validate_maker_payload` / `validate_checker_payload`，这些校验只能防止“结构错、映射错、缺项多项”，不能保证模型对业务规则的语义理解始终正确。模型升级、prompt 漂移、供应商 JSON 兼容性差异，都可能让相同规则得到不同的场景与评审结果。

### 风险二：人工 review 是质量闭环的一部分，也会成为吞吐瓶颈

我会按三阶段升级。

## 第一阶段：把 PoC 升成可试点工具

这一步先不追求全自动执行，先把现有设计链路做稳。

第一件事是把 **schema 升级成正式契约层**。现在仓库已经依赖 `atomic_rule`、`semantic_rule`、maker/checker 的严格结构和 required case types 来维持质量。下一步要把这些 schema 单独版本化，明确兼容策略，比如 `semantic_rule v1.0 -> v1.1` 允许新增字段但不破坏旧消费者；同时补一套 schema regression tests，保证换模型、换 prompt、换 provider 后，输出至少在结构和关键语义槽位上稳定。仓库当前已经把这些结构当主干了，所以这是最自然的升级起点。

第二件事是把 **LLM 输出质量控制从“只靠在线 validator”变成“离线评测 + 在线守门”**。现在的 maker/checker 机制已经有严格约束，但它们本质仍然是模型调用。升级时应该补一个 gold dataset：挑 50 到 200 条代表性规则，人工维护标准 `semantic_rule`、标准 scenario、标准 checker verdict。以后每次改 prompt、换模型、调 schema，都先跑回归分数，再决定是否上线。这一步能把“感觉变好了”变成“指标真的变好了”。仓库当前已经有 `artifacts/poc_two_rules` 作为样例入口，这正好可以扩成正式 benchmark。

第三件事是把 **文档抽取链路做成可审计的 ETL**。设计文档已经明确建议 `PDF -> pages -> clauses -> atomic_rule -> semantic_rule`，而且强调不要让模型直接从 PDF 出测试案例，因为覆盖口径和追溯会漂。升级时就该顺着这个思路做：每一层都持久化、带 hash、带 provenance、带 diff。这样以后如果 `semantic_rule` 变了，你能知道是 PDF 解析变了、clause 切分变了，还是语义归一化变了。

第四件事是把 **review-session 从本地工具变成团队可协作的审查服务**。仓库目前是 local HTTP review service，适合单机验证，不适合多人并发、权限控制和审计留痕。第一阶段不用上很重的平台化，只要补这几样就够：登录、角色权限、审查记录、决策历史、评论线程、任务分派、导出审计日志。这样它至少能支持一个测试设计小组试点。

## 第二阶段：补上真正的执行层

这是最关键的一跳。  
如果不做这层，它永远只是“AI 测试设计系统”。

最重要的是新增一个 **execution compiler**。它的职责不是让 LLM 直接写测试代码，而是把 `semantic_rule.execution_mapping`、`statement`、`test_design_hints` 编译成统一的执行中间表示，比如 `ExecutableScenario`。这个中间层里要有明确的前置条件、输入构造、动作、预期事件、断言、清理动作。因为设计文档已经把 `execution_mapping`、`dsl_assertions`、`observable_outputs` 这些槽位定义好了，所以最合理的升级路径就是：先编译成内部 IR，再由不同 runner 去执行。

然后要加 **runner adapter 层**。不要一开始就直接支持十种系统，先选你最需要的一类，例如：

- API / service 测试 runner  
- 消息 / FIX / event runner  
- UI runner  
- DB / audit / reconciliation runner

每个 runner 都实现同一套接口：`prepare -> execute -> observe -> assert -> cleanup`。这样这个 repo 才会从“测试设计”变成“测试设计可落地到多个执行域”。

再往下要补 **test environment orchestration**。现在仓库的流程停在 review 和 report，没有环境准备概念。执行层一进来，就必须有环境状态管理：测试账户、参考数据、市场时段、会话初始化、沙盒系统地址、依赖服务健康检查。没有这一层，再好的场景也只能停留在纸面。这个判断是基于当前公开流程中并未出现执行环境模块或运行时管理入口。

还要补 **oracle 层**。目前 checker 负责“场景设计质量与覆盖判断”，但执行之后需要另一类 oracle：真正判断系统行为是否符合规则。建议把 oracle 拆成三类：确定性断言、可计算断言、人工核查断言。能 deterministic 的绝不交给 LLM。比如字段格式、状态转换、时限规则、枚举值、计算结果，这些都应该尽量收敛成确定性校验器，而不是再用模型看一遍结果。

## 第三阶段：把它升成可治理的平台

这一步才是“长期可维护”。

第一，要做 **coverage 体系升级**。现在覆盖是按 `rule_type -> required_case_types` 做严格覆盖，这对设计层很好，但对执行层不够。升级后至少要有四层 coverage：

- document coverage：文档规则被抽到了多少  
- design coverage：规则被设计成场景了多少  
- executable coverage：场景里有多少能真正运行  
- pass/fail coverage：真正运行后通过了多少

这样你才能区分“设计完整”与“执行落地完整”。当前仓库只明显实现了前两层中的大部分。

第二，要做 **风险分级和路由**。不是所有规则都值得同样的 AI 比例。像 `enum_definition`、`data_constraint`、`deadline` 这类结构性强的规则，可以更快进入自动编译和自动执行；而 `workflow`、带歧义的复合条款、跨章节依赖规则，则保留更重的人审和更慢的上线节奏。设计文档已经把 `rule_type` 分得比较细，这正好可以变成升级路线里的自动化优先级。

第三，要做 **多模型与多供应商隔离**。仓库现在是 OpenAI-compatible provider adapter with retry support，这适合 PoC，但正式化时需要把“模型选择”从业务逻辑里进一步抽出去。maker、checker、rewrite、semantic normalization，最好拆成独立策略层，允许 A/B 测试、按 rule_type 路由模型、灰度发布、快速回滚。否则你一换模型，整个链路都在一起抖。

第四，要做 **可观测性**。不仅要记录日志，还要有指标。至少要追：

- 每层产物数量和失败率  
- 规则拆分漂移率  
- maker/checker 一致性  
- 人工 rewrite 比例  
- 可执行转化率  
- 执行通过率  
- 不同模型版本的回归差异

现在仓库已经有 terminal + file logging 和重试机制，但这还只是基础可靠性，不是平台级 observability。

## 我最建议的升级顺序

如果你想少走弯路，我建议按这个顺序做：

**先稳抽取，再稳 schema，再做 benchmark，再加执行 IR，再接一个最小 runner。**

原因很简单：  
如果前面的 `pages -> clauses -> atomic_rule -> semantic_rule` 不稳，后面执行层接得越多，返工越大。设计文档已经把这条分层链路当成核心原则了，所以升级时不要跳过。

更具体一点，6 个里程碑就够：

1. 固化 schema + 版本管理  
2. 建 benchmark + 回归评测  
3. 把 review-session 服务化  
4. 设计 `ExecutableScenario` 中间层  
5. 先接一个 runner，比如 API/FIX  
6. 补 execution report，把设计 coverage 和运行 coverage 打通

## 一句话建议

**不要把它升级成“更复杂的 maker/checker 系统”，要把它升级成“以 rule model 为核心、以 execution IR 为桥、以 deterministic oracle 为落点”的测试平台。**  
否则你只是把一个很聪明的测试设计原型，继续做成一个更贵的测试设计原型。

按**模板二**来：

## 一、系统定位

这个 repo 现在更像一个**测试设计系统**，不是完整的**测试执行系统**，也还谈不上真正的**测试编排平台**。README 直接把它定义为 “document-driven test design prototype”，当前主流程是：抽规则、归一化成 `atomic_rule` / `semantic_rule`、maker 生成结构化测试场景、checker 审查质量和覆盖、人工 review、rewrite、再出 HTML report。主入口也集中在 CLI、pipelines、review session、workflow session、reporting、provider adapter 这些设计与审核模块上。

它当前最容易被高估成“AI 自动化测试框架”。原因是 `semantic_rule` 已经有 `execution_mapping`、`observable_outputs`、`system_events`、`dsl_assertions` 这些字段，看起来很像能直接接执行层；但设计文档明确说 `semantic_rule` 是“语义层和半执行层”，要“对未来可执行测试保持足够接近，但不直接等于测试脚本”。也就是说，这些更像桥接结构，不是现成执行器。

它实际还缺的关键能力，是：真正的执行引擎、环境准备与编排、系统驱动层、运行时 oracle、执行结果回流与稳定的可观测性。这些在公开主流程和入口说明里都没有出现。

## 二、升级核心问题

### 1. 如何从“测试设计”补到“测试执行”

最关键的一跳，不是继续强化 maker/checker，而是新增一个**执行中间层**。  
建议把 `semantic_rule` 编译成统一的 `ExecutableScenario` 或类似 IR，里面固定表达：

- preconditions
- inputs / test data
- actions
- expected events / observable outputs
- deterministic assertions
- cleanup

这样 `execution_mapping` 和 `dsl_assertions` 才真正变成执行入口，而不是停留在表达意图。这个方向和仓库已有 rule model 是顺的，因为设计文档已经把 `execution_mapping` 定义成“连接未来可执行测试层的桥接字段”。

然后在 IR 下面接 **runner adapter**。不要一开始追求全能，先选一个最重要的执行域，例如 API / service，或者如果你的真实场景是 FIX，就先做 FIX runner。每个 runner 统一实现 `prepare -> execute -> observe -> assert -> cleanup` 接口。这样系统才会从“AI 设计测试”升级成“AI 驱动测试设计 + 可运行执行”。

### 2. 如何从“LLM 驱动”补到“确定性执行 + 确定性校验”

这个 repo 当前已经做了很多约束：maker 必须对每个输入规则返回恰好一个结果，每个 required case type 恰好一个 scenario；checker 必须对每个输入 case 返回恰好一个 review，且 `semantic_rule_id` 必须匹配。`pipelines.py` 还把不同 `rule_type` 映射成 required/optional case types，再直接传给 maker，避免模型自己猜。

但这些只是把 **LLM 输出收紧**，不是把系统变成确定性系统。升级时应该把判断拆成两类：

- **继续允许 LLM 做的事**：规则归一化、场景草拟、歧义提示、测试设计建议
- **必须确定性化的事**：字段格式校验、状态转换检查、时间窗口判断、计算结果校验、枚举合法性、覆盖统计、traceability 验证

尤其对 `data_constraint`、`deadline`、`calculation`、`enum_definition` 这类结构化规则，优先把 oracle 做成代码而不是再交给模型二次判断。因为仓库的 rule_type 已经分得很细，这正好可以作为“哪些先 deterministic 化”的切入点。

### 3. 如何从“本地工具/原型”补到“团队可协作系统”

README 里把 `review_session.py` 明确成 **local HTTP review session service**，页面可以保存 draft、提交 review、触发 `rewrite -> checker -> report`、finalize 并跳转报告。这很适合 PoC 和单机验证，但不适合团队级协作。

升级方向应该是把 review-session 服务化，至少补这几层：

- 身份与权限
- 审核任务分派
- 多人并发审查
- 审计日志与决策历史
- 评论线程 / 差异对比
- 版本冻结与回滚

也就是说，review 要从“本地页面”升级成“工作流系统的一环”。

### 4. 如何从“设计 coverage”补到“执行 coverage”

当前 repo 的 coverage 主要是**设计层 coverage**：一条规则只有在其所有 required case types 都被 checker 接受时，才算 fully covered。这个设计在测试设计阶段是合理的，而且 `pipelines.py` 已经把 `rule_type -> required_case_types` 机制固化了。

但平台化后至少要拆成四层：

- **document coverage**：文档规则有没有被抽出来
- **design coverage**：规则有没有被设计成场景
- **executable coverage**：设计出的场景里有多少能真正运行
- **runtime coverage**：运行后有多少通过，失败分布是什么

否则你只能知道“设计上好像覆盖了”，不知道“系统实际上有没有被测到”。

## 三、升级优先级

### 基础层：不做就无法继续升级的能力

**1. 固化 rule schema 与版本管理**  
现在 `atomic_rule` / `semantic_rule` 已经是整个系统的主干，设计文档还显式建议 `semantic_rule.version = 1.0`。下一步应该把 schema 抽成正式契约，单独版本化、做兼容规则和回归样本。否则一旦改 prompt、换模型、换字段，后面 maker/checker/execution 都会一起抖。

**2. 建立 benchmark 与 regression 评测**  
仓库现在推荐从 `artifacts/poc_two_rules/semantic_rules.json` 开始，说明它已有最小样例入口。下一步应该扩成正式 benchmark：人工维护一批 gold `semantic_rule`、gold scenarios、gold review 结果，每次改模型或 prompt 都先跑回归。

**3. 把文档抽取变成可审计 ETL**  
设计文档和脚本指南都强调 `PDF -> pages -> clauses -> atomic_rule -> semantic_rule` 的分层，并列出 `pages.json`、`clauses.json`、`atomic_rules.json`、`semantic_rules.json` 等中间产物。升级时要把这些中间层全部持久化、可 diff、可追溯。这样后续执行层一旦出问题，才能定位到底是 PDF 解析漂了，还是 semantic normalization 漂了。

这层做完，系统的质变是：从“容易漂的原型链路”变成“可追责、可回归的设计流水线”。

### 闭环层：让系统真正形成可运行闭环的能力

**1. 新增 execution IR / compiler**  
把 `semantic_rule` 编译成内部可执行中间表示，而不是直接让 LLM 产出最终脚本。这样可以把设计层和执行层隔开，减少后续 runner 扩展成本。证据基础是仓库已经有 `execution_mapping` 和 `dsl_assertions` 这些桥接字段。

**2. 先接一个最小 runner**  
建议先做一个垂直场景最强相关的 runner，而不是全栈支持。这样能尽快验证“从 rule 到 runnable scenario”这条主链路是否成立。

**3. 新增 deterministic oracle 层**  
checker 现在负责的是“场景设计质量与覆盖判断”，不是系统运行后的真 oracle。执行层必须单独引入确定性断言模块。仓库现有 checker 约束清楚说明了这一点：它返回的是结构化 blocking 与 coverage 字段，不是最终业务执行真值。

这层做完，系统的质变是：从“设计闭环”变成“设计 + 运行闭环”。

### 平台层：产品化、治理、可观测性、扩展性能力

**1. review-session 服务化**  
把 local HTTP 工具升级成团队协作服务。

**2. 多模型策略层**  
现在 provider 是 OpenAI-compatible adapter with retry support，适合 PoC。平台化时应把 maker/checker/rewrite 的模型选择独立出来，支持按 rule_type 路由、A/B、灰度、回滚。

**3. 可观测性与治理**  
现在有 logging 和 recoverable retry，但平台还需要更完整的指标：抽取漂移率、rule-to-scenario 转化率、checker block 率、人审重写率、可执行率、执行通过率。仓库当前只有基础日志和 retry，不足以支撑平台治理。

这层做完，系统的质变是：从“可跑的闭环”变成“可维护的平台”。

## 四、风险与取舍

最容易浪费时间的方向，是**继续堆 prompt 和 maker/checker 复杂度**，把它做成一个更贵的“测试设计系统”。因为它当前最大缺口不在设计层，而在执行层和确定性 oracle 层。仓库已经把设计层约束做得不算弱了：required case types、exactly-one constraints、schema validation、human review、retry/resume 都已经在。继续只加这层，收益会递减。

第二个不该太早做的，是“平台化大一统”。在还没有 execution IR 和最小 runner 前，就去做复杂权限、多租户、重前后端改造，容易把项目拖死。因为仓库当前连正式 release 都没有，主页显示只有 2 commits、0 stars、0 forks，而且 README 推荐从 two-rule POC 起步。现阶段更需要验证主链路，而不是先堆平台壳子。

第三个关键取舍是：**哪些能力该优先确定性化**。  
我会优先做：

- data constraints
- deadlines
- calculations
- enum validations
- traceability / coverage accounting

而 workflow、复杂歧义归纳、测试设计提示、自然语言 review comment 这类仍可保留 LLM 辅助。这个取舍直接利用了现有 rule_type 分类。

如果只能做一个最小版本，最该优先验证的是：

**`semantic_rule -> execution IR -> 一个 runner -> deterministic oracle -> execution report`**

只要这条链跑通，系统才真正迈出“从测试设计到测试执行”的那一步。

早期体验不错，扩到团队时才发现它其实不是协作系统。

# Enterprise AI Testing Upgrade Roadmap

## 1. Current Starting Point

The current system is a **document-driven test design prototype**. It already supports a usable design loop:

`source docs -> extract scripts -> atomic_rules.json -> semantic_rules.json -> maker -> checker -> review-session -> rewrite -> checker -> report` 

Today, the framework can already:
- extract rules from source documents into `atomic_rule` and `semantic_rule` artifacts, 
- generate BDD-style scenarios through `maker`, 
- assess scenario quality and rule coverage through `checker`, with strict coverage defined by `rule_type -> required_case_types`, 
- support human approval / rewrite / reject decisions in a local review session, 
- regenerate artifacts and produce final HTML reports. 

The current weaknesses are also clear:
- rule extraction governance is not presented as a first-class validation layer, 
- model and prompt stability governance are not explicit in the public workflow, 
- review is local and single-user oriented, 
- the repo still looks early-stage, with 2 commits and no published releases. 

## 2. Upgrade Principles

These principles apply across all phases.

### 2.1 Model outputs must be governed, not trusted
All LLM-driven steps must be treated as probabilistic. Every model-facing module must be constrained by:
- explicit schemas
- versioned prompts
- artifact metadata
- regression baselines
- rollbackable model configuration

### 2.2 Upstream rule quality controls downstream value
No downstream improvement can compensate for invalid `atomic_rule` or `semantic_rule` inputs. Upstream validation is mandatory before expanding downstream scope.

### 2.3 Separate deterministic responsibilities from LLM-assisted responsibilities
LLMs may assist with:
- rule normalization
- ambiguity detection
- test planning drafts
- BDD generation suggestions

Deterministic modules must own:
- schema validation
- enum validation
- coverage accounting
- traceability checks
- duplicate detection thresholds
- execution result assertions where possible

### 2.4 Every phase must define both scope and non-scope
Each phase must explicitly state what is included and what is deferred, to avoid uncontrolled expansion.

### 2.5 AI agents must work against contracts, not free-form instructions
Any AI coding agent contributing to the repo must operate against:
- versioned docs
- fixed acceptance criteria
- schema contracts
- test gates
- no direct undocumented structural changes

## 3. Cross-Model Governance Requirements

These rules must exist before or alongside any feature growth.

### 3.1 Provider abstraction
Keep the current OpenAI-compatible provider layer, but formalize it into a model strategy contract:
- provider name
- model name
- model version if available
- prompt version
- temperature / key decoding settings
- timeout / retry settings

All artifacts produced by maker, checker, rewrite, and future planner modules must record this metadata.

### 3.2 Prompt versioning
Every prompt used in production workflows must have:
- a stable prompt ID
- semantic version
- owner
- change log
- linked benchmark set

### 3.3 Regression baseline
The repo already recommends `artifacts/poc_two_rules/semantic_rules.json` as a fast validation entry point. That should become the first official baseline suite. 

Required baseline categories:
- extraction sanity
- maker structural validity
- checker consistency
- report generation
- end-to-end smoke

### 3.4 Model compatibility policy
Any new model API may only be adopted if it passes:
- schema conformance
- benchmark threshold
- checker stability threshold
- artifact diff review

### 3.5 Rollback policy
A new model or prompt version must be reversible without changing business logic or artifact schemas.

## 4.1 Stage Goal
Stabilize the current design pipeline so that different model APIs can be used without silently degrading artifact quality.

## 4.2 In Scope
- extraction validation
- schema formalization
- rule type governance
- baseline CI
- prompt / artifact versioning
- checker stability visibility
- repo documentation for AI agent execution

## 4.3 Out of Scope
- multi-user hosted review platform
- execution engine
- step definition integration
- advanced planning intelligence
- full enterprise workflow orchestration

## 4.4 Key Deliverables

### A. Formal schema layer
Add versioned JSON Schemas for:
- `atomic_rule`
- `semantic_rule`
- maker output
- checker output
- human review output

### B. Rule validation pipeline
Insert a formal validation stage:

`docs -> extraction scripts -> atomic_rules.json -> schema validation -> duplicate candidate detection -> rule_type enum validation -> semantic_rules.json`

### C. Minimal rule quality gates
Implement:
- required fields validation
- `rule_type` enum enforcement
- duplicate candidate detection
- invalid trace reference detection
- schema failure as hard-stop

### D. Prompt and artifact metadata
Every generated artifact must include:
- prompt version
- model ID
- provider
- run timestamp
- source artifact hash
- pipeline version

### E. Baseline CI
Add GitHub Actions to run:
- end-to-end smoke on `poc_two_rules`
- schema validation tests
- reporting smoke test
- core unit tests for pipeline and review session bootstrapping

### F. Checker stability signal
Run checker twice on the same small baseline and flag inconsistent conclusions.

### G. Repo docs for AI agents
Add:
- `docs/roadmap.md`
- `docs/acceptance.md`
- `docs/model_governance.md`
- `docs/agent_guidelines.md`

## 4.5 Acceptance Criteria
Short-term phase is accepted only if:
- all core artifacts are schema-validated in CI
- invalid `rule_type` values fail the pipeline
- baseline smoke run is reproducible in CI
- every maker/checker artifact records model and prompt metadata
- checker instability can be surfaced on the baseline set
- roadmap and governance docs are present in the repo

## 4.6 Success Metric
The system becomes **trustworthy enough for repeated internal development**, even if it is not yet enterprise-ready.

# 5. Mid-Term Plan (3–9 months)

## 5.1 Stage Goal
Upgrade from a stable design prototype into an AI-assisted **test planning and BDD generation platform** that can handle multiple document categories and produce reusable test design outputs.

## 5.2 In Scope
- document-type aware ingestion
- planning layer above maker/checker
- richer traceability
- improved review collaboration
- BDD normalization
- export interfaces for downstream execution teams

## 5.3 Out of Scope
- full runtime execution of all generated scenarios
- autonomous end-to-end execution against all systems
- full enterprise RBAC and production multi-tenant platform

## 5.4 Key Deliverables

### A. Multi-document ingestion framework
Support defined source classes, for example:
- rulebooks
- product specs
- API docs
- business workflow docs
- compliance / policy docs
- release notes / change documents

Each document class must have:
- parsing strategy
- extraction constraints
- expected rule patterns
- known failure modes

### B. Test planning layer
Introduce a planner stage between `semantic_rule` and BDD generation.

New flow:

`semantic_rules -> planning -> test objectives -> scenario families -> BDD generation`

The planner should produce:
- priority
- risk level
- coverage intent
- scenario family
- dependency notes
- recommended validation strategy

### C. Traceability model
Each final BDD scenario must trace back to:
- source document
- source clause / section
- atomic rule
- semantic rule
- planning decision
- checker verdict
- human review outcome

### D. BDD contract layer
Define a normalized BDD representation independent of final output syntax. This becomes the stable handoff artifact before step integration.

### E. Review collaboration v1
Add file-based or bundle-based review exchange:
- export review package
- import review package
- merge review decisions
- conflict surfacing

### F. Quality dashboards
Enhance reports with:
- rule_type coverage heatmap
- unstable checker decisions
- baseline drift versus previous runs
- document class breakdown
- traceability drill-down
- JSON/CSV export

## 5.5 Acceptance Criteria
Mid-term phase is accepted only if:
- at least 3 document classes are supported with explicit parsing rules
- planner outputs are versioned and schema-validated
- generated BDD artifacts are normalized and traceable
- review packages can be exported and merged
- report can show historical diffs and unstable judgments
- model change regression is enforced before adoption

## 5.6 Success Metric
The system becomes **usable for controlled team pilots** in document-driven test planning and BDD generation.

## 6.1 Stage Goal
Evolve the framework into an **enterprise AI testing platform** that connects document learning, planning, BDD generation, and execution integration with existing step definitions.

## 6.2 In Scope
- step definition mapping
- execution abstraction
- deterministic assertion layer
- hosted collaborative review service
- enterprise governance and observability
- controlled rollout across model providers

## 6.3 Out of Scope
- unrestricted autonomous test execution without approval gates
- fully free-form agentic code generation without schema and review controls

## 6.4 Key Deliverables

### A. Step definition integration layer
Create a mapping layer between normalized BDD steps and existing step definitions.

This should support:
- exact step match
- parameterized step match
- candidate step suggestion
- unmatched step reporting
- reuse score
- ownership per step library

### B. Execution readiness contract
Introduce an `ExecutableScenario` representation that extends BDD with:
- environment requirements
- input data requirements
- setup hooks
- expected deterministic assertions
- cleanup hooks
- linked step definitions

### C. Deterministic oracle framework
Move execution truth away from LLM judgment wherever possible.

Deterministic modules should own:
- field validation
- state validation
- calculation validation
- deadline/window checks
- event sequence verification
- pass/fail accounting

### D. Hosted review and governance service
Upgrade review-session from local HTTP service to deployable web service with:
- user accounts
- reviewer roles
- audit trail
- conflict resolution
- comment threads
- assignment workflow

### E. Enterprise observability
Track:
- ingestion failure rate
- extraction drift
- duplicate / conflict rule rate
- planner change rate
- maker validity rate
- checker instability rate
- BDD reuse rate
- step binding success rate
- execution pass/fail distribution

### F. Release governance
Formalize:
- release tags
- compatibility matrix
- benchmark gates
- migration notes
- approved provider list

## 6.5 Acceptance Criteria
Long-term phase is accepted only if:
- normalized BDD can be bound to existing step definitions with measurable reuse
- unmatched steps are surfaced automatically
- execution-ready scenarios can be exported consistently
- hosted review supports multi-user auditability
- deterministic assertions exist for at least the core structured rule categories
- provider rollout requires benchmark pass and rollback path
- enterprise-level metrics are available for audit and platform operation

## 6.6 Success Metric
The system becomes **enterprise-usable as an AI-assisted testing platform**, not just a design prototype.

## 一、两者的核心差异

### 1. 开发人员视角
前面基于 repo 形成的那套方案，本质上是**系统工程视角**。  
它关注的是：

- pipeline 怎么分层
- artifact contract 怎么定义
- schema、benchmark、CI、rollback 怎么治理
- roadmap 怎么分短中长期
- 模型 API 接入时如何保持架构稳定
- 未来怎么走到 BDD、step definition、execution contract

它的强项是：**架构清晰、阶段明确、工程治理比较强**。

### 2. 测试人员视角
你这次上传的 md，更像**测试资产治理与流程控制视角**。  
它关注的是：

- 提示词集合完整性，不能丢
- 更新记录必须完整
- 版本控制与回滚
- 提示词的执行顺序和依赖关系

- 并行执行、错误恢复、性能指标、增量更新

它的强项是：**测试过程控制细、审计意识强、对失败和变更更敏感**。

一句话说：

- **开发视角**更像在搭系统骨架
- **测试视角**更像在补质量神经和操作纪律

## 二、各自的优势和盲点

## A. 开发视角方案的优势

开发视角最大的优点有四个：

第一，它把系统边界讲清楚了。  
比如它明确区分了：

- 现在是 document-driven test design system
- 还不是 execution platform
- 未来要通过 normalized BDD、step mapping、ExecutableScenario 扩展

这对长期演进非常重要。

第二，它把**deterministic responsibilities** 和 **LLM-assisted responsibilities** 分开了。  
这在企业级 AI testing 里很关键，不然一切都会滑向“用模型多判一次”。

第三，它有比较清楚的**phase-based roadmap**。  
短期稳上游和治理，中期补 planning 和 BDD，长期再接 execution 和 step definitions，这个顺序是合理的。

第四，它天然适合写成 repo 文档和交给 AI agent 执行。

## B. 开发视角方案的盲点

它也有明显短板：

第一，**对测试资产生命周期的颗粒度不够细**。  
比如 prompt 集合完整性、编号连续性、更新记录模板、review 模板分类、失败分析模板，这些在开发视角里没有被当成一等公民。

第二，**对“人怎么审、怎么反馈、怎么积累失败经验”描述得还不够具体**。  
开发视角有 acceptance 和 review gate，但测试视角里那种 confidence、failure analysis、feedback template 更细。

第三，**对测试工件运营化的感觉没那么强**。  
开发视角更偏系统设计，而不是 test operations。

## C. 测试视角方案的优势

测试视角这份 md 的价值其实很高，尤其体现在三个地方。

第一，它把**提示词本身当成测试资产**来管理。  
这一点很强。  
不是把 prompt 当临时输入，而是当：
- 可编号
- 可追踪
- 可版本化
- 可回滚
- 可审查
- 有依赖关系

这对以后多模型 API 并行接入时非常有帮助。

第二，它天然带有**强验证文化**。  
你上传的 md 里经常出现：

- 执行前验证
- 执行后验证
- 集成测试步骤
- 预期结果
- 影响分析
- 回滚程序

这比普通 roadmap 更接近测试团队真正拿来落地的操作规范。

第三，它对**失败、恢复、性能、增量更新、并行执行**的敏感度更高。  
开发视角通常先关心“有没有这个模块”，而测试视角会追问：
- 失败了怎么办
- 并行时会不会乱
- 更新只改一部分时怎么验证
- 性能指标怎么收集

这些都很实用。

## D. 测试视角方案的盲点

它也有自己的问题。

第一，**过于 prompt-centric**。  
你这份 md 的组织中心是“15 个提示词的生命周期和执行顺序”，这很适合流程治理，但不一定适合长期架构演进。

第二，**容易把流程写得过重**。

第三，**对系统分层和模块边界不如开发视角清晰**。  
它很强于治理，但没有开发视角那样清楚地区分：
- provider layer
- schema layer
- review layer
- planning layer
- execution layer

最合理的方式不是“二选一”，而是：

**开发视角负责系统骨架，测试视角负责资产治理、验证纪律和审查机制。**

- 回滚程序

这些应该分别进入：
- `architecture.md`
- `acceptance.md`
- `model_governance.md`
- `agent_guidelines.md`

### 应该单独保留成测试治理文档的内容
测试视角里还有一部分内容，不适合硬塞进主架构文档，而应该单独成文：

- review template 分类体系

- `docs/testing_governance.md`
- `docs/prompt_lifecycle.md`

有些东西最好分层，不要揉成一份：

- **系统架构边界** 和 **测试操作模板**
- **模型治理规则** 和 **具体提示词文本**
- **长期 roadmap** 和 **某个 prompt 的执行脚本级细则**

否则文档会变得又重又难维护。

- 有 prompt 治理
