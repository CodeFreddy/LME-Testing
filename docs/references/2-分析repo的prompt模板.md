**专门分析 AI / testing / agent 类 repo**的 prompt 模板

## 版本一：通用强约束版

请分析这个 repo，但**不要只做功能介绍或 README 总结**。
我要的是**批判性架构审查**，重点看它的**局限性、风险、成熟度和能力边界**。

请把它当成一个“候选工程方案”来评估，回答下面问题：

### 1. 系统定位

* 这个 repo 本质上是什么系统？
* 它不是什么系统？
* 它属于测试设计、测试执行、测试编排、测试平台，还是其中某一层？

### 2. 已实现能力 vs 未实现能力

请明确区分：

* 哪些能力是代码和文档里已经真实落地的
* 哪些能力只是字段预留、接口预留、架构意图，实际上还没落地
* 哪些能力很容易被 README 叙事误导，以为“已经有了”

### 3. 核心局限

请重点分析：

* 这个系统最大的能力边界是什么？
* 它当前最不适合承担什么职责？
* 如果把它高估为完整平台，会错在哪里？

### 4. LLM / Agent 风险

请重点检查：

* 哪些关键环节依赖 LLM，而不是确定性算法？
* schema、prompt、retry、validator 这些机制，是“解决问题”还是“缓解问题”？
* 模型升级、prompt 漂移、厂商 JSON 兼容差异、上下文波动，会怎样影响结果？

### 5. 输入与抽取风险

如果 repo 涉及规则抽取、文档解析、RAG、PDF/Markdown 输入，请重点分析：

* 它对输入文档质量的依赖有多强？
* 哪些环节最容易引入误差？
* 对 PDF、表格、脚注、多栏排版、分页、章节边界等复杂情况是否脆弱？
* 作者有没有显式设计中间检查层来缓解这些问题？

### 6. 工程成熟度

请根据以下信号判断它更像：

* PoC
* 内部原型
* 可团队试点的工程工具
* 可正式落地的成熟项目

请结合这些证据：

* commit 历史
* release / tags
* CI / Actions
* 文档风格
* 样例规模
* 配置方式
* 目录结构
* 是否有测试、监控、部署、扩展机制

### 7. 结论必须批判

请不要平衡性地“优缺点各说一点”就结束。
我要你明确回答：

* 我**最容易高估**它的地方是什么？
* 我**最容易忽略**的风险是什么？
* 如果团队要采用它，**最大阻碍**是什么？
* 它现在更适合：

  * 方法验证
  * 小范围 PoC
  * 内部试点
  * 正式生产使用
    请明确选一个，并说明原因。

### 输出要求

* 不要只复述 README
* 优先找缺失项和未完成项
* 优先找结构性问题，不要给空泛风险
* 每个关键判断尽量给出 repo 里的具体证据
* 如果某个判断是推断，不是仓库直接证明，请明确标注“推断”
* 请优先寻找能推翻乐观判断的证据，而不是顺着作者叙事做总结
* 请重点分析缺失项、未闭环项、预留项，不要把“结构存在”误判为“能力已实现”
* 如果一个模块看起来像未来执行层/平台层的接口，请判断它到底是已落地能力，还是仅用于表达意图的桥接结构

### 输出格式

1. 一句话定位
2. 已落地能力
3. 容易被误以为已落地、其实没有的能力
4. 三个最大局限
5. 三个最高风险
6. 成熟度判断
7. 最终结论（一句话）
