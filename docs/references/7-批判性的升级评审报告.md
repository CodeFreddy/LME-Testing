**当前起点、最脆弱环节、治理缺口、silent failure、P0/P1/P2、长期演进。**

## 1. 当前起点

这个 repo 当前已经形成的真实流水线是：

**source docs → extract scripts → atomic_rules.json → semantic_rules.json → maker → checker → review-session → rewrite → checker → report**。README 还明确说它是一个 “document-driven test design prototype”，主目标是把 LME 文档转成结构化规则、生成 BDD 风格场景、做 checker 审核、支持人工 review/rewrite、再输出 HTML 报告。

已经明确存在的环节有：

* 规则抽取与中间产物：`atomic_rule`、`semantic_rule`。
* maker / checker / rewrite 主链。
* 本地 review-session。
* HTML report。
* provider retry / logging 这类基础可靠性措施。

还**没有被仓库公开证据明确证明**存在的环节有：

* 提取层的正式 schema 校验与冲突检测。
* 模型 / prompt 的版本治理与基线回归。
* checker 稳定性验证。
* 多人协作 review。
* 明确的 CI / 发布治理。
* 真正的执行层闭环。

这些不是说一定没有，而是就当前公开主页和 README 能看到的内容，没有被清楚呈现出来。

## 2. 最脆弱环节

**最脆弱环节是“原始文档 → atomic_rules.json / semantic_rules.json”这一层。**

原因不是它最显眼，而是它最上游，而且整个后链条都建立在它上面。README 把整个系统建立在规则抽取和归一化之上；一旦上游规则有漏抽、错拆、错类型，maker、checker、review、report 都会在错误输入上继续认真工作。

更关键的是，README 公开强调的是“抽取脚本”“归一化产物”“maker/checker/review/report”，但没有同样明确地展示“提取后的正式校验层”。这就意味着当前最危险的不是后面模块不工作，而是**上游错了，下游还能继续跑**。

所以你后来把重点放在提取层，是对的。这个判断比“先继续强化 maker/checker”更贴近仓库现状。

## 3. 三个最大治理缺口

### 缺口一：上游规则产物的治理不够显式

README 讲清了有 `atomic_rule`、`semantic_rule` 两层，但没有把“提取后自动 schema 校验、枚举约束、去重、冲突候选检测”明确成一个正式步骤。对于一个以规则为核心资产的系统，这意味着上游中间产物更像“约定正确”，而不是“系统性校验正确”。

这类缺口的危险是：
不是马上崩，而是 quietly 产出一套看起来很正式、其实根基已经偏了的下游结果。

### 缺口二：模型与 prompt 治理不够显式

README 公开展示了 provider adapter、retry support、推荐从 `poc_two_rules` 开始验证，但没有看到同等明确的“模型版本基线、prompt 版本、artifact metadata 记录、回归阈值”机制。也就是说，调用可靠性有一些缓冲，但**输出质量稳定性治理**没有同样强地呈现出来。

这会导致一个典型问题：
系统能跑，日志也正常，但 maker/checker 产物悄悄漂了，没有第一时间暴露。

### 缺口三：协作与工程保障偏原型化

README 明确把 `review_session.py` 定义为 **Local HTTP review session service**，而仓库主页显示目前只有 **2 commits**、**0 stars / 0 forks**，且没有 release 信号。这个组合更像单人/小范围原型使用，而不是已经具备多人协作与持续演进治理的团队工具。

这里我谨慎一点：
我不能仅凭主页就断言“完全没有 CI”，因为我没拿到 `.github/workflows` 的直接文件证据；但至少从当前公开信号看，成熟工程治理痕迹很弱。

## 4. 三个最大的 silent failure 风险

这是这次最重要的一段。

### 风险一：规则类型或规则拆分错了，但 maker/checker 还能继续跑

README 明确说 coverage 是按 `rule_type -> required_case_types` 做的，只有所有 required case types 被 checker 接受，才算 fully covered。这个逻辑本身没问题。问题在于：**如果上游 `rule_type` 本来就错了，后面的严格覆盖会在错误前提上认真计算。**

这就是典型 silent failure：
报表很完整，coverage 逻辑也严格，但严格地衡量了一套错规则。

### 风险二：checker 的结论看起来正式，但稳定性可能没有被单独治理

README 明确说 checker 会 assess case quality and rule coverage，且 human reviewer 决定 approve / rewrite / reject。也就是说，checker 在主链里是“正式审查者”角色。可如果 checker 自己也是模型驱动而缺少双跑一致性、基线对比、漂移告警，那它给出的“fully covered / not fully covered”就可能带着不稳定性，却在报告里呈现得像确定结论。

这也是 silent failure：
不是看不见结果，而是**结果太像真的**。

### 风险三：review-session 可用，但协作层面的缺口会在团队使用时才暴露

README 说明 review-session 能保存 draft、提交 review、触发 rewrite/checker/report、finalize，并跳到最终 report.html。这对单人验证很好。问题是，它同样明确是 local HTTP review service。也就是说，在单人阶段它“可用”，但多人异步审查、冲突合并、审计留痕这些问题，不会在早期暴露，而会在团队使用时突然放大。

这也是 silent failure：
早期体验不错，扩到团队时才发现它其实不是协作系统。

## 5. P0 / P1 / P2 改良项

这里我直接按“基于当前架构、最值得优先补的改良项”给。

### P0：先补可信度底座

第一，**规则提取后的正式校验层**。
把链路改成：

`文档 -> 提取脚本 -> atomic_rules.json -> schema校验 / rule_type枚举约束 / 重复规则候选 -> semantic_rules.json`

因为当前整个系统都是建立在这些规则产物上的。README 已经证明了这条主链存在，所以这里最值得优先补。

第二，**基础 CI + 冒烟回归**。
哪怕先只做最小版本，也应该至少让 `poc_two_rules` 成为一个自动可跑的冒烟用例。README 已经明确把它作为推荐起点。

第三，**artifact 元数据治理**。
每次 maker/checker 运行，都记录模型、prompt 版本、时间戳、输入规则集版本。
这一步是为了防止“结果变了但没人知道为什么”。

### P1：补模型稳定性治理

第一，**maker 回归基线**。
用 `poc_two_rules` 或小规模金标准集，做最小回归比较。README 已经把 `poc_two_rules` 定位成快速验证入口，非常适合作为基线样本。

第二，**checker 双跑一致性检测**。
先别急着把 checker 改得更复杂，先测它同批输入是否稳定。
因为当前最大风险不是 checker 没功能，而是它的结论可能被默认成过于确定。

第三，**报告里分离 coverage 与 stability**。
不要只显示 fully/partially covered，建议另加一个稳定性维度，例如 stable / unstable。这样不会和现有 coverage 主逻辑打架，同时能把“不确定但看起来完整”的问题显出来。这个建议是基于 README 已明确存在 coverage 主逻辑的推断性扩展。

### P2：补协作与分析能力

第一，**review 结果导出/导入与合并**。
当前 README 已明确有 draft 保存和 finalize，但还停在本地 review service 形态。先补异步导出/导入，比直接重平台化更稳。

第二，**报告增强**。
增加 rule_type 热力图、历史对比、JSON/CSV 导出。
这些不会改变主链，但会明显提升“发现问题”的效率。当前 README 已证明报告和多视图存在，所以这属于自然增量。

第三，**traceability 可视化**。
把 source docs / atomic_rule / semantic_rule / scenario / checker verdict 串起来。这会比单纯漂亮报表更有价值，因为这个系统本身就是文档驱动规则链路。

## 6. 长期演进方向

这里和 P0/P1/P2 分开讲。

如果你现在的目标是**先把现有流水线做稳**，那上面的改良项已经足够。
但如果看长期，下一步还是会走到两个方向：

第一，**把设计可信度进一步工程化**。
也就是把你现在提出的 schema、枚举、回归、checker 稳定性、review 协作全做实。

第二，**再决定是否往执行层演进**。
也就是以后要不要从“设计系统”扩到“执行系统”。
但这一步不该抢在现在前面，因为当前最明显的问题还在现有链路的可信度，而不是能力野心不够大。这个判断是基于仓库现状做的优先级建议，不是说执行层不重要。

## 7. 一句话结论

**基于当前仓库公开状态，这个系统最该优先补的不是更复杂的 AI 能力，而是“上游规则产物校验 + 模型/结论稳定性治理 + 最小工程保障”；因为它当前最大的风险不是跑不起来，而是能跑、能出报告、但结果可能 silently 不够可信。**
